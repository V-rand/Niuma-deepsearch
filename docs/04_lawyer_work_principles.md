# 律师案件工作的原则性理解（供 Codex / Agent 使用）

## 1. 这不是固定流水线，而是长期专业服务系统

律师代理案件不是固定步骤执行，而是围绕一个案件长期持续地维护事实、证据、程序、沟通和结果。

案件会因为法院通知、客户反馈、对方动作、裁判结果、执行结果而不断变化，因此系统必须支持动态规划，而不是死板流程图。

## 2. 一个案件就是一个长期 session

案件往往持续数月甚至数年。系统不应把案件理解为一次性任务，而应理解为：
- 有独立上下文；
- 有独立文件目录；
- 有独立待办；
- 有独立期限和事件；
- 有可长期演化的状态。

## 3. 文书和文件是核心工作载体

律师工作的核心成果并不是“聊天结论”，而是：
- 起诉状
- 答辩状
- 上诉状
- 申请书
- 证据清单
- 策略报告
- 庭审简报
- 代理词
- 执行申请书
- 结案小结
- 归档卷宗

所以系统必须以文件系统为中心，而不是只在内存里保存上下文。

## 4. 事实永远优先于漂亮结论

律师工作必须不断追问：
- 事实是什么；
- 哪些事实已被证据支持；
- 哪些事实仍待核实；
- 哪些说法只是客户陈述而未被证据固化；
- 哪些事实之间存在冲突。

系统不应过早给出结论，而应先维护事实层和证据层。

## 5. 检索必须同时保留摘要与原始来源

法律检索贯穿整个案件生命周期。每次检索都应至少保留：
- 查询内容；
- 关键结论摘要；
- 原始检索结果 / 原文；
- 来源路径和追溯信息；
- 该检索对当前策略的影响。

不要只保留结论性摘要，否则长期运行中会失真。

## 6. 期限管理是硬约束，不是装饰功能

民事诉讼中的很多动作都受期限约束，例如：
- 答辩期限
- 交费期限
- 举证期限
- 上诉期限
- 再审期限
- 执行期限
- 庭前准备时限
- 保全后的起诉时限

系统必须主动管理期限、自动提醒、生成待办，并在期限临近时强制唤醒主 Agent。

## 7. 律师工作是持续沟通，不是只对法院输出

律师不仅向法院提交材料，也持续与客户沟通：
- 案情理解
- 风险提示
- 材料补充
- 文书确认
- 庭审安排
- 裁判结果
- 上诉 / 执行 / 再审选择
- 费用和发票
- 结案与归档

因此系统要把“沟通事件”视为案件核心状态的一部分。

## 8. 每个阶段都可能反复迭代

案件分析可能反复补证；
庭前准备可能因新证据重新调整；
庭后可能因法官要求继续补正；
执行可能因财产线索变化反复推进；
再审和执行也可能交错。

所以系统不应假设阶段只会单向前进一次。

## 9. 主 Agent 负责规划，不应被 workflow 写死

主 Agent 应根据：
- 当前阶段
- 当前事实状态
- 当前证据状态
- 当前期限风险
- 当前外部事件
- 当前客户决策

来决定下一步动作。

skill 应提供策略和边界，tool 应提供能力，但底层 kernel 不应写死业务流程。

## 10. sub-agent 是助手，不是第二个大脑

如需拆分任务，可创建 sub-agent 处理局部任务，例如：
- 整理某批材料
- 检索某类法规案例
- 归档某轮庭后文件
- 生成某类阶段摘要

但最终决策、阶段判断、策略变更和 session 状态维护应由主 Agent 完成。

## 11. session memory 应围绕 markdown 文件组织

推荐长期维护以下几类 memory：
- session 总览
- 时间线
- 稳定事实
- 待核实问题
- 当前策略
- 当前待办
- 检索记录
- 庭审记录
- 裁判结果
- 执行进展
- 结案总结
- 全局演进日志

这些内容应尽量落到 `.md`，并允许长期追溯和人工接管。

## 12. 律师工作的目标不是“说服模型”，而是“推动案件”

系统最终服务的是案件推进和结果落地。

因此所有推理都应尽量转化为：
- 新待办
- 新文书
- 新检索
- 新沟通动作
- 新状态更新
- 新期限跟踪

而不是停留在抽象讨论层面。
